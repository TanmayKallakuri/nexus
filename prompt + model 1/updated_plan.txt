ARCHITECTURE: ML-AUGMENTED LLM DOPPELGANGER
============================================

The LLM is the prediction engine. ML models feed it
sharper context. This gives us the best of both worlds:
LLM handles unseen questions, ML prevents mean-regression.

  ┌────────────────────────────────────────┐
  │         PRE-COMPUTED (before test)     │
  │                                        │
  │  Model 1: Person Response Profiles     │
  │  → response style, construct scores,   │
  │    cognitive performance, game behavior │
  │                                        │
  │  Model 2: Behavioral Clustering        │
  │  → cluster 233 people into ~5-8 types  │
  │  → label each cluster behaviorally     │
  │                                        │
  │  Model 3: Question-Construct Mapping   │
  │  → embed all known questions           │
  │  → map new questions to constructs     │
  │  → pull person's scores on that area   │
  └──────────────────┬─────────────────────┘
                     │
                     ▼
  ┌────────────────────────────────────────┐
  │         AT PREDICTION TIME             │
  │                                        │
  │  LLM Prompt =                          │
  │    persona text file                   │
  │    + Model 1 response profile          │
  │    + Model 2 cluster label             │
  │    + Model 3 construct match           │
  │    + new question + answer options     │
  │                                        │
  │  LLM → predicted answer               │
  └────────────────────────────────────────┘

============================================================
KEY METRICS TO TRACK
============================================================

1. MAE (Mean Absolute Error) — how close on average?
2. Person-wise correlation — does the model distinguish
   between individuals? THIS IS CRITICAL.
3. Variance ratio — predicted variance / actual variance.
   If < 0.8, model is collapsing to the mean.
4. Exact match rate — for categorical questions.
5. Calibration error — are predicted distributions realistic?

============================================================
VALIDATION STRATEGY
============================================================

LEAVE-OUT-QUESTION validation:
- For each person, randomly hold out 20% of their questions
- Predict held-out questions using only persona text + ML signals
- This simulates the hidden test perfectly

LEAVE-OUT-CONSTRUCT validation:
- Hold out an entire construct (e.g., all Risk Aversion questions)
- Predict using everything EXCEPT that construct
- This tests true generalization to unseen question types

Run BOTH. Report both in the slides.

============================================================
RED FLAGS — STOP IF YOU SEE THESE
============================================================

- Predicted variance across people < 50% of actual variance
  → model is averaging everyone, correlation will tank
- MAE barely changes when you switch between different people
  → model is ignoring the persona, treating everyone the same
- Validation accuracy on held-out constructs much worse than
  held-out questions from known constructs
  → model is memorizing, not generalizing
- LLM frequently outputs the midpoint (3 on 1-5 scale)
  → prompt is too vague, needs stronger persona signals

============================================================
LLM API CHOICE
============================================================

Recommended: Claude API (anthropic package)
  - Best at following complex persona instructions
  - Strong reasoning about psychological profiles
  - Use claude-sonnet-4-20250514 for speed during validation
  - Use claude-opus-4-0-20250115 for final predictions if budget allows

Alternative: OpenAI GPT-4o
  - Also strong, slightly cheaper per token
  - Good fallback if rate-limited

For 233 people x ~50 test questions = ~11,650 API calls.
At ~2000 tokens per call, that's ~23M tokens.
Budget accordingly.
