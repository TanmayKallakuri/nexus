CHANGE 3: CALIBRATION LAYER (VARIANCE STRETCHING)
Priority: HIGH — do this after retraining with changes 1+2
============================================================

WHY:
  Even after changes 1 and 2, LightGBM will still somewhat
  regress to the mean. This post-hoc step stretches predictions
  to match the actual variance, without retraining.

HOW:

  After you get predictions from the ensemble, apply this
  PER QUESTION (not globally):

  def calibrate_predictions(test_df, stretch_factor=None):
      """
      Stretch predictions away from per-question mean
      to match actual variance.
      """
      calibrated = test_df.copy()

      for qid in calibrated['question_id'].unique():
          mask = calibrated['question_id'] == qid
          preds = calibrated.loc[mask, 'predicted'].values

          q_mean = preds.mean()
          q_pred_std = preds.std()

          if q_pred_std < 0.01:
              # No variance to stretch — skip
              continue

          if stretch_factor is None:
              # Auto-compute: target std ~1.1 (observed average across training)
              target_std = 1.1
              sf = target_std / q_pred_std
          else:
              sf = stretch_factor

          # Stretch
          stretched = q_mean + sf * (preds - q_mean)

          # Clamp to valid range
          num_opts = calibrated.loc[mask, 'num_options'].iloc[0]
          stretched = np.clip(stretched, 1, num_opts)

          calibrated.loc[mask, 'predicted'] = stretched

      return calibrated

  # Tune stretch_factor on validation:
  # Try values: 1.5, 2.0, 2.5, 3.0, 3.5
  # Pick the one that maximizes Pearson r on holdout questions

  best_r = -1
  best_sf = 1.0
  for sf in [1.5, 2.0, 2.5, 3.0, 3.5]:
      cal_df = calibrate_predictions(test_df, stretch_factor=sf)

      # Compute Pearson r on calibrated predictions
      correlations = []
      for qid in holdout_qids:
          q_mask = cal_df['question_id'] == qid
          actual = cal_df.loc[q_mask, 'answer_position'].values
          predicted = cal_df.loc[q_mask, 'predicted'].values
          if len(actual) >= 3 and np.std(actual) > 0 and np.std(predicted) > 0:
              from scipy.stats import pearsonr
              r, _ = pearsonr(actual, predicted)
              correlations.append(r)
      mean_r = np.mean(correlations)

      # Also compute accuracy
      cal_df['abs_error'] = np.abs(cal_df['predicted'] - cal_df['answer_position'])
      cal_df['range'] = cal_df['num_options'] - 1
      accuracy = 1 - (cal_df['abs_error'].mean() / cal_df['range'].mean())

      print(f"  stretch={sf}: accuracy={accuracy:.4f}, r={mean_r:.4f}")
      if mean_r > best_r:
          best_r = mean_r
          best_sf = sf

  print(f"\nBest stretch factor: {best_sf} (r={best_r:.4f})")

CHANGE 4: BUILD PERCENTILE BASELINE AND BLEND
Priority: MEDIUM — do this after changes 1-3
============================================================

WHY:
  A dead-simple model that maps person construct percentile
  directly to the answer scale has PERFECT variance by
  construction. Blending it with LightGBM corrects for
  variance collapse.

HOW:

  def percentile_baseline_predict(person_id, question_id, num_options,
                                   construct_mapping, person_construct):
      """
      Simple predictor: person percentile on matched construct → answer
      """
      # Find construct for this question
      cm = construct_mapping[construct_mapping['question_id'] == question_id]
      if cm.empty:
          return num_options / 2  # midpoint fallback

      construct_id = cm['construct_id'].iloc[0]

      # Find person's percentile on this construct
      pc = person_construct[
          (person_construct['person_id'] == person_id) &
          (person_construct['construct_id'] == construct_id)
      ]
      if pc.empty:
          return num_options / 2  # midpoint fallback

      percentile = pc['percentile'].iloc[0]

      # Map percentile to answer scale: 1 to num_options
      prediction = 1 + percentile * (num_options - 1)
      return prediction

  # Compute percentile baseline for all test rows
  test_df['percentile_pred'] = test_df.apply(
      lambda row: percentile_baseline_predict(
          row['person_id'], row['question_id'], row['num_options'],
          construct_mapping, person_construct
      ), axis=1
  )

  # Blend with LightGBM
  # Try different weights on validation
  best_r = -1
  best_w = 0.5
  for w_lgbm in [0.3, 0.4, 0.5, 0.6, 0.7, 0.8]:
      test_df['blended'] = w_lgbm * test_df['predicted'] + (1 - w_lgbm) * test_df['percentile_pred']

      correlations = []
      for qid in holdout_qids:
          q_mask = test_df['question_id'] == qid
          actual = test_df.loc[q_mask, 'answer_position'].values
          predicted = test_df.loc[q_mask, 'blended'].values
          if len(actual) >= 3 and np.std(actual) > 0 and np.std(predicted) > 0:
              r, _ = pearsonr(actual, predicted)
              correlations.append(r)

      mean_r = np.mean(correlations)
      print(f"  w_lgbm={w_lgbm}: r={mean_r:.4f}")
      if mean_r > best_r:
          best_r = mean_r
          best_w = w_lgbm

  print(f"\nBest blend: {best_w} LightGBM + {1-best_w} percentile (r={best_r:.4f})")

EXPECTED IMPACT:
  Correlation improves because percentile baseline has natural
  person-level spread that combats LightGBM's compression.

============================================================
CODEX REVIEW - SAFER V2 IMPLEMENTATION PLAN
Use this plan instead of following the original V2 literally
============================================================

BOTTOM LINE:
  The V2 ideas are directionally good, but there is a major
  validation risk:

  - construct features can leak held-out answers if they are
    built from the full dataset before the holdout split
  - many constructs are actually single-question constructs,
    so "person percentile on this construct" can become a
    disguised copy of the target if we are not careful

  Therefore:
  DO NOT trust any validation result unless person-level
  construct features are rebuilt inside each fold using only
  training questions.

============================================================
WHAT WE LEARNED FROM V1
============================================================

V1 metrics:
  Accuracy:       0.724
  Mean Pearson r: 0.259
  Variance ratio: 0.288

Diagnosis:
  1. The model is collapsing toward the mean
     - predicted std is far below actual std
     - this hurts Pearson r even when MAE looks acceptable

  2. One model for all question types is too coarse
     - Personality works best
     - Economic preferences is negative
     - Cognitive and Demographics are weak

  3. Raw answer_position is not the right target for every item
     - true ordinal Likert items behave differently from
       binary/categorical knowledge items

============================================================
REVISED BUILD ORDER - DO THIS INSTEAD
============================================================

ROUND 0 - FIX VALIDATION FIRST
---------------------------------
Before any new modeling:

  1. Split holdout by question_id
  2. Build all person-level aggregate features using ONLY the
     training questions in that fold
  3. For multi-item constructs, compute leave-one-item-out
     construct features so the target item is not included in
     the person's own aggregate

Why this matters:
  If held-out answers are used to build person aggregates,
  validation is optimistic and feature importance is misleading.

============================================================
ROUND 1 - PERSONALITY-ONLY LEAKAGE-FREE MODEL
Highest-priority model
============================================================

Goal:
  Build a strong model for Personality questions first, because
  that is the only block already showing real signal.

Features:
  - Model 1 person features rebuilt inside each fold
  - question text embeddings
  - num_options
  - block_name / question_type
  - leave-one-item-out construct features for the matched scale:
      * person_construct_mean_loo
      * person_construct_std_loo
      * person_construct_percentile_loo

Target:
  Use normalized ordinal target instead of raw answer_position:

    y_norm = (answer_position - 1) / (num_options - 1)

  This makes 5-point and 6-point items comparable and reduces
  scale compression.

Expected outcome:
  Personality Pearson r should improve first.
  Do not optimize other blocks until this works.

============================================================
ROUND 2 - SPLIT NON-PERSONALITY BY QUESTION TYPE
Do not force one regressor to do every task
============================================================

Recommended routing:

  A. Personality matrix / Likert items
     -> regression or ordinal model on normalized target

  B. Binary economic choices
     -> classifier or probability model, not raw-position
        regression

  C. Cognitive multiple-choice / true-false items
     -> classifier, not ordinal regression on answer_position

  D. Demographics
     -> separate classifier or exclude from the main Tolendi
        correlation model if they continue to drag results down

Why:
  The current V1 model treats synonym questions, Likert scales,
  binary economic decisions, and employment status as one
  unified numeric task. That is structurally wrong.

============================================================
ROUND 3 - REDUCE QUESTION EMBEDDING DIMENSION
Regularization step, not the first fix
============================================================

Apply PCA to question embeddings AFTER the leakage fix:

  - try 25, 50, and 100 dimensions
  - keep the smallest value that preserves good validation

Why:
  V1 uses 384 question embedding dimensions versus 92 person

  to "who is this person."

Note:
  50 dimensions is a reasonable starting point, but not a rule.

============================================================
ROUND 4 - ONLY THEN TRY CALIBRATION
Calibration cannot rescue weak ranking
============================================================

Use variance stretching only after:
  - leakage-free validation is in place
  - ranking quality is already decent
  - mean Pearson r is clearly improved

Guideline:
  If calibration improves variance ratio but not Pearson r,
  it is cosmetic and should not be trusted.

============================================================
ROUND 5 - BASELINE + BLENDING, WITH SAFETY CHECKS
============================================================

Percentile baseline can be useful, but only with caution.

Rule:
  Do NOT use percentile baseline blindly for all constructs.

Reason:
  Many constructs are single-question constructs.
  For those items, construct percentile can behave like a
  near-direct encoding of the target if computed from the full
  data.

Safe use:
  - use percentile baseline only for true multi-item constructs
  - compute it from training-only data
  - compare against the ML model honestly on holdout questions

If the percentile baseline alone beats the ML model:
  That is acceptable, but it means the mapping is doing most of
  the work and the ensemble is not yet earning its complexity.
